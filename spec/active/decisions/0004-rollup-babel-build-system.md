# ADR-0004: 使用 Rollup + Babel 构建 ES5 兼容脚本

**状态**: 🟢 已采纳
**日期**: 2021-01-01
**决策者**: Cooper
**替代者**: N/A

## 背景
Tampermonkey 用户脚本需要在各种浏览器环境中运行，包括较旧的浏览器（如 IE11）。同时，为了提高开发效率，希望使用现代 JavaScript 语法（ES6+）和模块化开发。需要选择合适的构建工具链。

## 决策
我们决定使用 Rollup 作为打包工具，Babel 作为转译工具，将 ES6+ 模块化代码打包并转译为 ES5 兼容的单文件脚本。

**构建配置**：
- **Rollup**: 将 `src/` 下的 ES6 模块打包成单个文件
- **Babel**: 使用 `@babel/preset-env` 转译为 ES5 语法
- **目标**: IE11 兼容性
- **输出**: `dist/雀魂金玉四麻风格分析助手-v{version}.user.js`

## 理由

### 模块化开发
- Rollup 支持 ES6 模块，可以将代码拆分为多个文件
- 便于代码组织和维护（game/, api/, analysis/, ui/ 等模块）
- 支持 tree-shaking，自动移除未使用的代码

### 兼容性保障
- Babel 可以将 ES6+ 语法转译为 ES5
- `@babel/preset-env` 自动处理目标环境的兼容性
- 确保脚本在 IE11 等旧浏览器中正常运行

### 单文件输出
- Tampermonkey 脚本必须是单个文件
- Rollup 可以将多个模块打包成一个文件
- 输出文件包含所有依赖，无需外部加载

### 开发体验
- 支持 watch 模式，自动重新构建
- 构建速度快，适合迭代开发
- 配置简单，易于维护

## 备选方案

### 方案 A: Webpack + Babel
- 优点：
  - 功能强大，生态丰富
  - 支持各种资源类型（CSS、图片等）
- 缺点：
  - 配置复杂，学习曲线陡峭
  - 打包体积较大（包含 Webpack 运行时）
  - 对于简单的脚本项目过于重量级
- 为何未采纳：复杂度过高，不适合单文件脚本项目

### 方案 B: 直接使用 Babel
- 优点：
  - 配置简单
  - 只做转译，不做打包
- 缺点：
  - 无法处理模块化（需要手动合并文件）
  - 无法 tree-shaking
  - 需要额外工具处理文件合并
- 为何未采纳：无法支持模块化开发

### 方案 C: 不使用构建工具，直接写 ES5
- 优点：
  - 无需构建步骤
  - 兼容性最好
- 缺点：
  - 无法使用现代 JavaScript 语法
  - 无法模块化开发，所有代码在一个文件中
  - 开发效率低，代码难以维护
- 为何未采纳：开发体验差，不利于长期维护

### 方案 D: Parcel
- 优点：
  - 零配置，开箱即用
  - 自动处理依赖和转译
- 缺点：
  - 对 Tampermonkey 脚本支持不佳
  - 输出文件可能包含不必要的运行时代码
  - 定制化能力较弱
- 为何未采纳：不适合 Tampermonkey 脚本的特殊需求

## 后果

### 正面影响
- 可以使用现代 JavaScript 语法（ES6+）开发，提高开发效率
- 代码模块化，易于组织和维护
- 自动转译为 ES5，确保兼容性
- 单文件输出，符合 Tampermonkey 要求
- 支持 watch 模式，开发体验良好

### 负面影响
- 需要构建步骤，增加了开发流程的复杂度
- 需要维护构建配置（rollup.config.js、.babelrc）
- 构建错误可能导致脚本无法运行
- 需要安装 Node.js 和 npm 依赖

### 权衡
我们选择了开发效率和代码可维护性，接受了构建步骤带来的额外复杂度。对于长期维护的项目，这是值得的。

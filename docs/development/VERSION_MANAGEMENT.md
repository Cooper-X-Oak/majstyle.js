# 版本管理规范

## 版本号规则

采用语义化版本号（Semantic Versioning）：`vMAJOR.MINOR.PATCH[-PRERELEASE]`

### 版本号组成

- **MAJOR（主版本号）**: 不兼容的 API 修改或重大架构变更
- **MINOR（次版本号）**: 向下兼容的功能性新增
- **PATCH（修订号）**: 向下兼容的问题修正
- **PRERELEASE（预发布标识）**: 可选，用于测试版本

### 预发布标识

- `alpha`: 内部测试版，功能不完整
- `beta`: 公开测试版，功能基本完整但可能有 bug
- `rc`: 候选发布版，准备正式发布

### 示例

- `2.0.0` - 正式版
- `2.1.0-beta.1` - 2.1.0 的第一个 beta 测试版
- `2.1.0-beta.2` - 2.1.0 的第二个 beta 测试版
- `2.1.0-rc.1` - 2.1.0 的候选发布版
- `2.1.0` - 2.1.0 正式版

## 版本发布流程

### 1. 开发阶段

- 在 `src/` 目录开发新功能
- 版本号使用 `X.Y.Z-beta.N`
- 每次构建生成 `dist/雀魂金玉四麻风格分析助手-vX.Y.Z-beta.N.user.js`

### 2. 测试阶段

- 安装 beta 版本到 Tampermonkey
- 在实际游戏中测试
- 修复 bug，递增 beta 版本号（beta.1 → beta.2 → beta.3）
- 每个 beta 版本都归档到 `archive/v2.x/`

### 3. 候选发布阶段（可选）

- 功能稳定后，发布 `X.Y.Z-rc.1`
- 最后一轮测试
- 如有问题，递增 rc 版本号

### 4. 正式发布

- 移除预发布标识，发布 `X.Y.Z`
- 更新 CHANGELOG.md
- 归档到 `archive/v2.x/`
- 打 git tag（如果使用 git）

## 文件管理

### 目录结构

```
majstyle.js/
├── src/                          # 源代码
├── dist/                         # 构建产物（当前版本）
│   └── 雀魂金玉四麻风格分析助手-vX.Y.Z.user.js
├── archive/                      # 历史版本归档
│   ├── v1.x/                     # 1.x 系列
│   ├── v2.x/                     # 2.x 系列
│   │   ├── 雀魂金玉四麻风格分析助手-v2.0.0.user.js
│   │   ├── 雀魂金玉四麻风格分析助手-v2.1.0-beta.1.user.js
│   │   ├── 雀魂金玉四麻风格分析助手-v2.1.0-beta.2.user.js
│   │   └── 雀魂金玉四麻风格分析助手-v2.1.0.user.js
│   └── pro-series/               # Pro 系列（已废弃）
├── CHANGELOG.md                  # 更新日志
├── VERSION_MANAGEMENT.md         # 本文档
└── package.json                  # 版本号配置
```

### 归档规则

1. **每个版本都要归档**（包括 beta 版本）
2. **归档时机**：构建新版本前，先归档当前版本
3. **归档位置**：`archive/vX.x/`（按主版本号分类）
4. **文件命名**：保持原文件名，包含完整版本号

### 归档命令

```bash
# 归档当前版本
cp dist/雀魂金玉四麻风格分析助手-v2.1.0-beta.1.user.js archive/v2.x/

# 构建新版本
npm run build
```

## 版本更新检查清单

发布新版本前，确保完成以下步骤：

- [ ] 更新 `package.json` 中的 `version`
- [ ] 更新 `rollup.config.js` 中的 `metadata.version` 和 `output.file`
- [ ] 归档当前版本到 `archive/vX.x/`
- [ ] 运行 `npm run build` 构建新版本
- [ ] 更新 `CHANGELOG.md`，记录变更内容
- [ ] 在实际游戏中测试新版本
- [ ] 如果是正式版，更新 README.md 中的版本号和下载链接

---

## 回滚策略

如果新版本有严重问题：

1. 从 `archive/vX.x/` 恢复上一个稳定版本
2. 在 CHANGELOG.md 中标记问题版本为 "已撤回"
3. 修复问题后，递增版本号重新发布

---

## 注意事项

1. **测试版本不要标记为正式版**
2. **每个版本都要归档**，即使是测试版
3. **版本号只增不减**，不要重复使用版本号
4. **CHANGELOG.md 要及时更新**，记录每个版本的变更
5. **正式版发布前要充分测试**，至少经过 2-3 个 beta 版本
